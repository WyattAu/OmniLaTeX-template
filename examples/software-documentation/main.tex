\begin{abstract}
DataPipe is an open-source streaming ETL (Extract, Transform, Load) platform
designed for real-time data integration.  This document provides a comprehensive
guide covering installation, configuration, core concepts, the plugin API, and
troubleshooting procedures.
\end{abstract}

\tableofcontents

\section{Overview}

DataPipe ingests data from heterogeneous sources, applies configurable
transformation pipelines, and delivers processed records to downstream systems
with sub-second latency.  The architecture is built on Apache Kafka for
transport and Apache Flink for stateful stream processing.

Key characteristics:

\begin{itemize}
    \item \textbf{Throughput:} Sustained ingestion of 500\,000 events per second
          on a three-node cluster.
    \item \textbf{Latency:} End-to-end processing latency under 200\,ms at P95.
    \item \textbf{Durability:} At-least-once delivery with exactly-once semantics
          available for supported sinks.
    \item \textbf{Extensibility:} A Java/Scala plugin API for custom connectors
          and transformation functions.
\end{itemize}

\section{Installation}

\subsection{Quick Start with Docker Compose}

The fastest way to run DataPipe locally is via the provided Compose file:

\begin{verbatim}
git clone https://github.com/datapipeline/datapipe.git
cd datapipe
docker compose -f docker-compose.dev.yaml up -d
\end{verbatim}

This starts a single-node Kafka broker, a Flink JobManager and TaskManager,
and the DataPipe web UI on port 8080.

\subsection{Production Installation}

For production environments, deploy DataPipe on a Kubernetes cluster:

\begin{enumerate}
    \item Add the Helm repository:
    \begin{verbatim}
helm repo add datapipe https://charts.datapipeline.io/stable
helm repo update
\end{verbatim}

    \item Create a values file (\texttt{values-prod.yaml}) with your Kafka
          bootstrap servers, PostgreSQL connection string, and resource limits.

    \item Install the release:
    \begin{verbatim}
helm install datapipe datapipe/datapipe \
    -n datapipe --create-namespace \
    -f values-prod.yaml
\end{verbatim}

    \item Verify the deployment:
    \begin{verbatim}
kubectl get pods -n datapipe
kubectl logs -f deployment/datapipe-web -n datapipe
\end{verbatim}
\end{enumerate}

\section{Configuration}

Configuration is managed through environment variables and a YAML file
(\texttt{datapipe.yaml}).  The most important settings are:

\begin{table}[htbp]
    \centering
    \caption{Core configuration parameters.}
    \label{tab:config}
    \begin{tabular}{@{}lp{6cm}l@{}}
        \toprule
        \textbf{Parameter} & \textbf{Description} & \textbf{Default} \\
        \midrule
        \texttt{kafka.brokers}       & Comma-separated Kafka broker addresses
            & \texttt{localhost:9092} \\
        \texttt{kafka.group\_id}     & Consumer group identifier
            & \texttt{datapipe-default} \\
        \texttt{flink.parallelism}   & Default Flink job parallelism
            & \texttt{4} \\
        \texttt{flink.checkpoint\_interval} & Checkpoint interval in milliseconds
            & \texttt{60000} \\
        \texttt{pg.url}              & PostgreSQL JDBC connection URL
            & --- \\
        \texttt{pg.username}         & PostgreSQL username
            & --- \\
        \texttt{pg.password}         & PostgreSQL password (use secrets manager)
            & --- \\
        \bottomrule
    \end{tabular}
\end{table}

Example \texttt{datapipe.yaml}:

\begin{verbatim}
kafka:
  brokers: "kafka-1:9092,kafka-2:9092,kafka-3:9092"
  group_id: "datapipe-prod"
flink:
  parallelism: 16
  checkpoint_interval: 30000
  state_backend: "rocksdb"
pg:
  url: "jdbc:postgresql://db.internal:5432/datapipe"
  username: "${DB_USER}"
  password: "${DB_PASSWORD}"
logging:
  level: "INFO"
  format: "json"
\end{verbatim}

\subsection{gRPC Ingestion API}

The gRPC ingestion service accepts protocol buffer messages for maximum
throughput.  Define your schema in \texttt{ingestion.proto}:

\begin{verbatim}
syntax = "proto3";
package datapipe.ingestion;

service IngestionService {
    rpc Ingest (stream Record) returns (IngestResponse);
    rpc IngestBatch (stream IngestBatchRequest)
        returns (IngestResponse);
}

message Record {
    string source = 1;
    int64 timestamp = 2;
    map<string, string> payload = 3;
}
\end{verbatim}

\section{Troubleshooting}

\subsection{Common Issues}

\begin{description}
    \item[Kafka connection refused] Verify that Kafka brokers are running and
          the \texttt{kafka.brokers} configuration matches your cluster
          addresses.  Check firewall rules and security group settings.

    \item[High checkpoint duration] If checkpoints consistently exceed the
          configured interval, increase \texttt{flink.checkpoint\_timeout} or
          reduce state size by tuning window intervals.

    \item[Out of memory errors] Increase the JVM heap size via
          \texttt{FLINK\_TASKMANAGER\_MEMORY} or
          \texttt{JAVA\_OPTS=-Xmx16g}.  Monitor GC logs for excessive
          full garbage collections.

    \item[Schema evolution failures] When a source connector schema changes,
          update the pipeline definition before restarting.  DataPipe supports
          backward-compatible schema evolution by default.
\end{description}

\subsection{Health Checks}

DataPipe exposes health endpoints for integration with load balancers and
orchestrators:

\begin{itemize}
    \item \texttt{/health/ready} --- Returns 200 when the service can accept
          traffic.
    \item \texttt{/health/live} --- Returns 200 when the service process is
          alive.
    \item \texttt{/metrics} --- Prometheus-format metrics endpoint.
\end{itemize}
